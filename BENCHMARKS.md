# Performance parity — go-ndarray vs NumPy

_Generated by `benchmarks/gen_report.sh` — every number is measured on this machine, none hand-edited._

## Methodology

- **Machine**: Apple M4 Max, 16 logical cores, macOS (host, not a VM).
- **Go**: go1.26.4, `CGO_ENABLED=0`, `GOWORK=off`; `go test -bench -benchtime=2s`. go-ndarray runs multi-core (default `GOMAXPROCS`).
- **NumPy**: 2.2.4, BLAS backend **accelerate unknown** (Apple's tuned vecLib). Two columns:
  - **numpy 1-thread** — `OPENBLAS_NUM_THREADS=OMP_NUM_THREADS=MKL_NUM_THREADS=VECLIB_MAXIMUM_THREADS=1`: the fair core comparison (go-ndarray's own kernels vs single-threaded NumPy C/BLAS).
  - **numpy default** — NumPy's normal multi-threaded BLAS: the real-world gap, especially for matmul.
- **gonum** (MatMul table only): gonum's pure-Go BLAS-backed `mat.Dense.Mul` (gonum v0.15.1), measured on this host — the pure-Go peer go-ndarray must beat. The MatMul before→after block and its gonum column are measured back-to-back under matched thermal state and are the one hand-curated section; the live go-ndarray numbers there are still those `benchmarks/gen_report.sh` reproduces.
- ns/op: Go reports its own; NumPy is min-of-7 over many inner iters (`benchmarks/bench_numpy.py`).
- Throughput: **GFLOP/s** = `2·M·N·K / t` for matmul/dot, **GB/s** = `bytes_touched / t` for the memory-bound element-wise / reduction / view ops.
- **Correctness gate**: VERIFY OK: 56 checks agree with NumPy (rel tol 1e-12) (`benchmarks/verify_numpy.py`, run before timing — the table is only published if go-ndarray matches NumPy numerically).

`ratio` = numpy-1-thread ns ÷ go ns. **ratio ≥ 1.0 ⇒ go-ndarray meets or beats single-threaded NumPy.** Verdict thresholds: ≥0.95 parity/win, 0.75–0.95 close, <0.75 lags.

## 2026-06-22 optimization pass (algorithmic) — BEFORE → AFTER

This pass closed the worst gaps, all of which were algorithmic (naive strided gathers, missing bulk `copy()`, single-threaded axis reductions, and a single-column GEMM standing in for mat·vec / dot) rather than a want of SIMD. #### 2026-06-23 GEMM-parallelism pass — BEFORE → AFTER

The packed/blocked GEMM and its SIMD micro-kernel (NEON 4×8 FMLA on arm64, SSE2 4×4
on amd64, scalar 4×4 elsewhere) were already in place and the single-core kernel was
near its ceiling (~57 GFLOP/s/core, L1-resident). The remaining gap was **parallel
efficiency**, not the kernel: the old `MatMulP` gave each worker the whole blocked
loop, so every worker re-packed the entire B panel (up to 1 MiB) GOMAXPROCS times per
contraction block — O(P) redundant memory traffic — and split the rows into one static
band per core, which on the M4's 12 P + 4 E asymmetric cores made every fast P-core
stall at the join on the slow E-cores (measured: 16 cores was *slower* than 10 on
1024²). Two changes fixed it, both bit-exact (the ikj summation order is unchanged):

1. **Pack B once, share it.** B is packed a single time per (jc, pc) cache block into a
   shared read-only buffer; workers only pack their own (cheap) A panel.
2. **Dynamic work-stealing bands.** The rows are diced into many MR-aligned bands and
   workers pull them off an atomic counter, so the P-cores absorb more bands than the
   E-cores instead of waiting on them.

#### 2026-06-23 small-N parallel-threshold pass — BEFORE → AFTER

A follow-up pass tuned **where** the GEMM crosses from serial to parallel. The single
fan-out threshold (`GemmThreshold`, m·n result elements) was shared between MatMul and
the bandwidth-bound MatVec and sat at 1<<14 = 16,384 — sized for MatVec, far too high
for the compute-bound GEMM. It left the whole 80²–120² band (the shapes that fit a few
cores' worth of work) stuck on the slow single-core path. The two thresholds were
**decoupled**: `GemmThreshold` lowered to the measured GEMM crossover (6,000) and
MatVec kept on its own `matVecThreshold` (1<<14, since forking a few-thousand-element
mat·vec is a net loss — 100×100 mat·vec is ~2.5µs serial vs ~5.8µs parallel). 32²/64²
stay serial (where the single-core register-blocked kernel is fastest); 80²+ now fans
out. Bit-exact — only the serial/parallel dispatch boundary moved, not the arithmetic.
