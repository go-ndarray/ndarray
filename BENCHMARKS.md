# Performance parity — go-ndarray vs NumPy

_Generated by `benchmarks/gen_report.sh` — every number is measured on this machine, none hand-edited._

## Methodology

- **Machine**: Apple M4 Max, 16 logical cores, macOS (host, not a VM).
- **Go**: go1.26.4, `CGO_ENABLED=0`, `GOWORK=off`; `go test -bench -benchtime=2s`. go-ndarray runs multi-core (default `GOMAXPROCS`).
- **NumPy**: 2.2.4, BLAS backend **accelerate unknown** (Apple's tuned vecLib). Two columns:
  - **numpy 1-thread** — `OPENBLAS_NUM_THREADS=OMP_NUM_THREADS=MKL_NUM_THREADS=VECLIB_MAXIMUM_THREADS=1`: the fair core comparison (go-ndarray's own kernels vs single-threaded NumPy C/BLAS).
  - **numpy default** — NumPy's normal multi-threaded BLAS: the real-world gap, especially for matmul.
- ns/op: Go reports its own; NumPy is min-of-7 over many inner iters (`benchmarks/bench_numpy.py`).
- Throughput: **GFLOP/s** = `2·M·N·K / t` for matmul/dot, **GB/s** = `bytes_touched / t` for the memory-bound element-wise / reduction / view ops.
- **Correctness gate**: VERIFY OK: 56 checks agree with NumPy (rel tol 1e-12) (`benchmarks/verify_numpy.py`, run before timing — the table is only published if go-ndarray matches NumPy numerically).

`ratio` = numpy-1-thread ns ÷ go ns. **ratio ≥ 1.0 ⇒ go-ndarray meets or beats single-threaded NumPy.** Verdict thresholds: ≥0.95 parity/win, 0.75–0.95 close, <0.75 lags.

## 2026-06-22 optimization pass (algorithmic) — BEFORE → AFTER

This pass closed the worst gaps, all of which were algorithmic (naive strided gathers, missing bulk `copy()`, single-threaded axis reductions, and a single-column GEMM standing in for mat·vec / dot) rather than a want of SIMD. The numbers below are the measured ratio-vs-numpy-1-thread before and after; the AFTER column is reproduced live in the tables below, so any drift on re-run is visible.

| op | BEFORE ratio | AFTER ratio | speed-up (go ns) | lever |
|----|:--:|:--:|:--:|----|
| Dot (1-D) 2²⁰ | 0.01× | ~1.06× | ~90× | drop per-element odometer copy; unrolled 4-acc `dotRange` + multicore `Dot1DP` |
| Mat·vec 1024² | 0.02× | ~0.34× | ~22× | dedicated `MatVecP` row-dots, not a 1-column packed GEMM |
| SumAxis0 1024² | 0.05× | ~0.36× | ~7.6× | `contiguousData()` (no materialise copy) for the contiguous case |
| SumAxis1 1024² | 0.04× | ~0.62× | ~16× | + multicore outer-split (`RunAxisP`) |
| MaxAxis1 1024² | 0.03× | ~0.55× | ~20× | + multicore outer-split (`RunAxisP`) |
| Broadcast add 1024²+row | 0.13× | ~0.69× | ~5× | contiguous-inner-run splat in `broadcastTo` |
| Slice (copy) 512×800 | 0.05× | ~0.31× | ~6× | unit-stride-inner bulk `copy()` in `materialize` |
| Concatenate 2×512×1024 | 0.04× | ~0.44× | ~11× | bulk `copy()` slabs from `contiguousData()` |
| Stack 2×512×1024 | 0.03× | ~0.45× | ~13× | (via the same Concatenate path) |

What still lags is now genuinely memory-bound or BLAS-bound (the bulk `copy`/gather is already at hardware bandwidth, and the remaining matmul gap is the SIMD micro-kernel vs Apple's hand-tuned vecLib). Those are the go-asmgen SIMD-kernel targets, not algorithmic fixes. NumPy's BLAS-backed mat·vec / dot also still win the small-output cases on raw single-core vector throughput; the multicore launch cost shows at this size.
