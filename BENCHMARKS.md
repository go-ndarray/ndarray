## Methodology

- **Machine**: Apple M4 Max, 16 logical cores, macOS (host, not a VM).
- **Go**: go1.26.4, `CGO_ENABLED=0`, `GOWORK=off`; `go test -bench -benchtime=2s`. go-ndarray runs multi-core (default `GOMAXPROCS`).
- **NumPy**: 2.2.4, BLAS backend **accelerate unknown** (Apple's tuned vecLib). Two columns:
  - **numpy 1-thread** вЂ” `OPENBLAS_NUM_THREADS=OMP_NUM_THREADS=MKL_NUM_THREADS=VECLIB_MAXIMUM_THREADS=1`: the fair core comparison (go-ndarray's own kernels vs single-threaded NumPy C/BLAS).
  - **numpy default** вЂ” NumPy's normal multi-threaded BLAS: the real-world gap, especially for matmul.
- **gonum** (MatMul table only): gonum's pure-Go BLAS-backed `mat.Dense.Mul` (gonum v0.15.1), measured on this host вЂ” the pure-Go peer go-ndarray must beat. The MatMul beforeв†’after block and its gonum column are measured back-to-back under matched thermal state and are the one hand-curated section; the live go-ndarray numbers there are still those `benchmarks/gen_report.sh` reproduces.
- ns/op: Go reports its own; NumPy is min-of-7 over many inner iters (`benchmarks/bench_numpy.py`).
- Throughput: **GFLOP/s** = `2В·MВ·NВ·K / t` for matmul/dot, **GB/s** = `bytes_touched / t` for the memory-bound element-wise / reduction / view ops.
- **Correctness gate**: VERIFY OK: 56 checks agree with NumPy (rel tol 1e-12) (`benchmarks/verify_numpy.py`, run before timing вЂ” the table is only published if go-ndarray matches NumPy numerically).

`ratio` = numpy-1-thread ns Г· go ns. **ratio в‰Ґ 1.0 в‡’ go-ndarray meets or beats single-threaded NumPy.** Verdict thresholds: в‰Ґ0.95 parity/win, 0.75вЂ“0.95 close, <0.75 lags.

#### 2026-06-23 GEMM-parallelism pass вЂ” BEFORE в†’ AFTER

The packed/blocked GEMM and its SIMD micro-kernel (NEON 4Г—8 FMLA on arm64, SSE2 4Г—4
on amd64, scalar 4Г—4 elsewhere) were already in place and the single-core kernel was
near its ceiling (~57 GFLOP/s/core, L1-resident). The remaining gap was **parallel
efficiency**, not the kernel: the old `MatMulP` gave each worker the whole blocked
loop, so every worker re-packed the entire B panel (up to 1 MiB) GOMAXPROCS times per
contraction block вЂ” O(P) redundant memory traffic вЂ” and split the rows into one static
band per core, which on the M4's 12 P + 4 E asymmetric cores made every fast P-core
stall at the join on the slow E-cores (measured: 16 cores was *slower* than 10 on
1024ВІ). Two changes fixed it, both bit-exact (the ikj summation order is unchanged):

1. **Pack B once, share it.** B is packed a single time per (jc, pc) cache block into a
   shared read-only buffer; workers only pack their own (cheap) A panel.
2. **Dynamic work-stealing bands.** The rows are diced into many MR-aligned bands and
   workers pull them off an atomic counter, so the P-cores absorb more bands than the
   E-cores instead of waiting on them.
